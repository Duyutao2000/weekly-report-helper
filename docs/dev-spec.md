# 开发文档

## 1. 项目概述

个人周报助手，本地待办事项管理工具，支持每周任务追踪、下周计划管理、一键生成周报。

- **目标用户**：单人本地使用
- **运行形态**：桌面 GUI 应用（Python + Gradio，浏览器本地访问）
- **数据存储**：本地 SQLite 单文件

## 2. 核心功能

### 2.1 项目管理

| 功能 | 说明 |
|------|------|
| 新增项目 | 输入项目名称，创建后自动排到列表末尾 |
| 编辑项目 | 修改项目名称 |
| 删除项目 | 删除项目，同时删除该项目下所有任务和计划（需二次确认） |
| 查看项目 | 按创建时间升序排列（最早创建的在前），无默认项目 |

**约束**：
- 项目名称不可为空、不可重复（trim 后比较）
- 删除项目时弹窗二次确认："该项目下有 N 条任务和 M 条计划，确认删除？"

### 2.2 任务管理

| 功能 | 说明 |
|------|------|
| 新增任务 | 选择项目、填写任务描述、填写工时（0.5h 精度） |
| 编辑任务 | 修改项目、描述、工时（状态不变） |
| 完成任务 | 标记为已完成，记录完成时间 |
| 撤销完成 | 将已完成任务恢复为 pending，清空完成时间 |
| 删除任务 | 软删除（`is_deleted = true`），不在列表中显示但保留数据 |
| 查看任务 | 当前周：按项目分组；历史任务：独立页面，支持筛选 |

**约束**：
- 任务描述不可为空
- 工时必须是 0.5 的整数倍（0.5, 1, 1.5, ...）
- 编辑/删除仅限 pending 状态的任务；已完成任务不可编辑，只能撤销后重新编辑

### 2.3 周期管理

| 功能 | 说明 |
|------|------|
| 默认周期 | 当前自然周（周一 ~ 周五，5 个工作日） |
| 自定义周期 | 手动选择起止日期，天数不限 |
| 切换周期 | 当前视图切换到目标周期，历史任务不受影响 |
| 历史任务 | 独立页面，按周筛选展示所有历史任务 |

**核心语义**："切换周期"是视觉重置——仅改变当前视图显示的周范围。所有任务按 `week_start`/`week_end` 持久化存储，切换周期只是更换了查询条件，数据不会丢失。

**下周计划同步检测**：切换周期时，自动检测目标周是否有未同步的上周计划（见 2.5）。

### 2.4 工时管理

| 规则 | 说明 |
|------|------|
| 默认总工时 | 40 小时/周（5 工作日 × 8 小时） |
| 自定义总工时 | 提供入口修改本周总工时（如遇节假日、请假），修改仅影响当前周 |
| 分配方式 | 每个任务手动填写工时 |
| 剩余计算 | 实时计算：`剩余 = 总工时 - 已分配工时总和`（包含 pending + completed） |
| 硬约束 | 总工时 40h（或自定义值）为硬约束 |
| 超时提示 | 已分配工时超过总工时时，界面上醒目提示（红色警告），但**不阻止添加**——用户可自行调整 |

**工时精度**：0.5h（6分钟）为最小单位。输入时将小数四舍五入到最近的 0.5。

**自定义工时入口**：在主界面工时段落旁放置编辑图标，点击可修改当前周总工时。默认 40h，修改后仅对当前周期生效，切换周期后恢复默认值。

### 2.5 下周计划

**设计意图**：当周填写下周计划 → 下周打开应用时自动提醒同步 → 一键转为待办任务，无需重复填写。

| 功能 | 说明 |
|------|------|
| 新增计划任务 | 选择项目、填写任务描述（**无工时**——计划阶段不分配工时） |
| 编辑/删除计划 | 仅限未同步的计划，已同步的只读 |
| 生成计划文本 | 调用大模型润色输出（格式化 + 书面化） |
| 同步到本周任务 | 将目标周 = 当前周的计划复制到 `tasks` 表 |

**目标周选择**：计划区域有一个目标周选择器，**默认值自动 = 当前周 + 1 周**。用户可手动修改（如遇节假日，跳过假期周，直接规划节后第一周）。切换主周期时，目标周自动跟随（当前周 + 1），手动修改过的值会被覆盖。

**示例——节假日场景**：
```
当前主周期: 2026-09-28 ~ 2026-10-02 (国庆前，正常上班)
计划目标周默认: 2026-10-05 ~ 2026-10-09 (国庆假期，不办公)
用户手动改为: 2026-10-12 ~ 2026-10-16 (节后第一周) ✓
→ 10/12 切换周期时，系统检测到目标周 10/12 有未同步计划 → 弹窗提醒
```

**同步检测时机**：每次切换周期时触发。逻辑如下：

1. 用户切换到某个周（目标周）
2. 查询 `next_week_plans`：`week_start = 目标周开始日期 AND synced_at IS NULL`
3. 如果存在未同步的计划，弹窗："发现 N 条来自上周的计划，是否同步为本周任务？"
4. 用户确认 → 每条计划复制为 `tasks` 行（status=pending, hours=NULL, week_start/end=目标周）
5. 批量更新 `next_week_plans.synced_at = NOW()`
6. 用户拒绝 → 不做任何操作，计划保留在 `next_week_plans`，下次切换仍会提示

**数据模型关键字段**：`next_week_plans.week_start` / `week_end` 存储的是**目标周**（计划所属的周），而非创建时的当前周。这样查询逻辑简单、可追溯。

**输出格式**：
```
[项目名称]

1. [任务描述1]
2. [任务描述2]
```

### 2.6 周报生成

**触发方式**：用户手动点击"生成周报"按钮。

**前置校验**：
1. 当前周无已完成任务 → **报错**："本周没有已完成的任务，无法生成周报"，停止流程
2. 当前周已完成任务存在 → 继续

**处理流程**：
1. 从 `tasks` 表查询当前周已完成任务（`status=completed AND week_start/end 匹配`）
2. 按项目分组，汇总每个项目工时
3. 构建 LLM prompt（任务列表 + 格式要求 + 书面化润色指令）
4. 调用大模型 API → 重试策略见下
5. 成功后保存到 `reports` 表，关联 `task_ids`（JSON 数组，记录参与生成的任务 ID）
6. 在界面上展示生成结果，提供"复制"和"重新生成"按钮

**输出格式**：
```
[项目名称] 工时：XX小时

1. [任务描述1]
2. [任务描述2]

[下一个项目...]
```

**任务关联**：`reports` 表通过周时间范围（`week_start`/`week_end`）与任务建立隐含关联，同时 `task_ids` 字段记录生成快照时的具体任务 ID 列表，确保可追溯。

**LLM 调用重试策略**：

```
                     ┌─ 调用 LLM API ─┐
                     │                │
                     ▼                ▼
                 成功 ✅          失败 ❌
                                    │
                           判断是否值得重试
                                  │
                  ┌───────────────┴───────────────┐
                  ▼                               ▼
            值得重试                          不值得重试
         (429 / 5xx /                        (400 / 401 /
          网络超时)                          403 / 413)
                  │                               │
         指数退避 + 随机抖动                      ▼
         最多重试 3 次                       直接报错
         总超时 30s                          展示原因
```

| 错误类型 | HTTP 状态码 / 条件 | 是否重试 | 原因 |
|---------|-------------------|---------|------|
| 频率限制 | 429 | ✅ 重试 | 等待后恢复 |
| 服务端错误 | 5xx | ✅ 重试 | 临时故障 |
| 网络超时 | timeout / connection reset | ✅ 重试 | 网络抖动 |
| 请求错误 | 400 | ❌ 不重试 | prompt 有问题 |
| 认证错误 | 401 / 403 | ❌ 不重试 | API Key 问题 |
| 内容过长 | 413 | ❌ 不重试 | 需缩减输入 |

**退避算法**：指数退避 + 完全抖动（Full Jitter）

```
base = 1 秒
max_delay = 16 秒
delay = min(max_delay, base × 2^attempt) × random(0, 1)
```

| 重试次数 | 延迟范围 |
|---------|---------|
| 第 1 次 | 0 ~ 2s |
| 第 2 次 | 0 ~ 4s |
| 第 3 次 | 0 ~ 8s |

超过 3 次或总耗时超过 30s → 终止并报错。

### 2.7 计划生成

**触发方式**：用户手动点击"生成计划"按钮。

**前置校验**：
1. `next_week_plans` 中当前目标周无数据 → 报错："请先添加下周计划"，停止流程
2. 有数据 → 继续

**处理逻辑**：与周报生成共享同一个 LLM 服务层，单次 API 调用完成。重试策略同 2.6。

## 3. 数据模型

### 3.1 项目表 (projects)

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| name | VARCHAR(100) | NOT NULL, UNIQUE | 项目名称（trim 后存储） |
| created_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 最后修改时间 |

**排序**：`ORDER BY created_at ASC`（最早创建的在前）

### 3.2 任务表 (tasks)

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| project_id | INTEGER | NOT NULL, FK → projects.id | 关联项目 |
| description | TEXT | NOT NULL | 任务描述 |
| hours | FLOAT | CHECK(hours >= 0 AND hours % 0.5 = 0) | 工时，0.5h 精度 |
| status | VARCHAR(20) | NOT NULL, DEFAULT 'pending', CHECK(status IN ('pending','completed')) | 任务状态 |
| week_start | DATE | NOT NULL | 所属周开始日期 |
| week_end | DATE | NOT NULL | 所属周结束日期 |
| is_deleted | BOOLEAN | NOT NULL, DEFAULT 0 | 软删除标记 |
| created_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 最后修改时间 |
| completed_at | DATETIME | NULL | 完成时间（pending 时为 NULL） |

**索引**：
- `idx_tasks_week ON tasks(week_start, week_end)`
- `idx_tasks_project ON tasks(project_id)`
- `idx_tasks_status ON tasks(status)`

**注意**：SQLite 不支持真正的 DECIMAL，`hours` 使用 FLOAT。应用层在读写时做 0.5 精度舍入，避免浮点累计误差影响工时总计显示。

### 3.3 下周计划表 (next_week_plans)

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| project_id | INTEGER | NOT NULL, FK → projects.id | 关联项目 |
| description | TEXT | NOT NULL | 任务描述 |
| week_start | DATE | NOT NULL | **目标周**开始日期（计划所属的周） |
| week_end | DATE | NOT NULL | **目标周**结束日期 |
| synced_at | DATETIME | NULL | 同步时间（NULL = 未同步） |
| created_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 最后修改时间 |

**关键设计**：`week_start`/`week_end` 是目标周（计划打算在哪周执行），不是创建时的当前周。`synced_at` 替代原 BOOLEAN `synced` 字段，可精确追踪同步状态和时间。

**索引**：
- `idx_plans_week ON next_week_plans(week_start, week_end)`
- `idx_plans_synced ON next_week_plans(synced_at)`

### 3.4 周报表 (reports)

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| week_start | DATE | NOT NULL | 周开始日期 |
| week_end | DATE | NOT NULL | 周结束日期 |
| content | TEXT | NOT NULL | LLM 生成的周报内容 |
| task_ids | TEXT | NOT NULL | JSON 数组，记录生成时使用的任务 ID（如 `[1,2,5]`） |
| created_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 生成时间 |

**任务关联**：通过 `week_start`/`week_end` 建立隐含关联（同一周的已完成任务），`task_ids` 提供精确快照（生成瞬间的具体任务集合）。

**索引**：
- `idx_reports_week ON reports(week_start, week_end)`

### 3.5 设置表 (settings)

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| key | VARCHAR(100) | PK | 配置键 |
| value | TEXT | NOT NULL | 配置值 |
| updated_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 最后修改时间 |

**预置数据**：

| key | value | 说明 |
|-----|-------|------|
| total_hours_per_week | 40 | 每周默认总工时 |
| llm_model | qwen-plus | 大模型标识 |
| llm_base_url | https://dashscope.aliyuncs.com/compatible-mode/v1 | API 地址 |

**自定义周工时**：用户通过任务看板"应用"按钮修改后，**写入 `settings.total_hours_per_week` 持久化**，重启后仍保留。切换周期时自动重置为该默认值。

### 5.7 数据库初始化

应用启动时自动执行：

1. 创建 `data/` 目录（如不存在）

### 5.9 日志

使用 Python `logging` 标准库，输出到 `data/app.log`：

- **INFO**：应用启动、周报生成、计划同步等关键操作
- **WARNING**：工时超限、LLM 重试等非阻塞异常
- **ERROR**：LLM 调用失败、数据库错误等需要关注的异常

日志格式：`%(asctime)s [%(levelname)s] %(name)s: %(message)s`
